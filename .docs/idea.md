I started this project to build a proper, reproducible Shona speech data pipeline instead of one-off curation. I ingested the labeled WaxalNLP `sna_asr` splits, normalized metadata and text, ran audio cleanup/quality scoring, and pushed the cleaned dataset to Hugging Face as `manassehzw/sna-dataset`. That was the first big milestone: a general-purpose dataset with provenance columns and auditable processing steps.

After that, I tested manual speaker quality/identity review using pulled speaker samples. I realized manual ear-testing at scale is too slow and still uncertain because contamination can hide outside small samples. I also hit practical issues (preemptions, zip corruption, salvage/repackaging), which made me formalize tooling for checkpointing, integrity checks, and local audit workflows.

My priority has now shifted from immediate premium TTS curation to speaker-label reliability. I want to produce a second-pass labeled variant (`sna-dataset-labeled`) where speaker IDs are reassigned by embedding-based clustering, with confidence and traceability. The goal is not perfect real-world identity truth, but consistent-enough speaker grouping to reduce label noise during downstream TTS fine-tuning. Premium TTS subset selection is intentionally deferred until this relabeling pass is solid.

Running the first clustering audit (v2) with ECAPA-TDNN + HDBSCAN + a Wav2Vec2-based gender model revealed a deeper problem: the source gender labels in WaxalNLP are unreliable, and the off-the-shelf gender model (trained on Common Voice — predominantly English and European speech) was producing confident wrong predictions on Shona speakers. The clearest example was a known high-quality female speaker where the model called 11 out of 18 clips Male. That was a majority. The result was spurious MIXED_GENDER clusters — not because those speakers were ambiguous, but because the gender model didn't generalise.

Rather than swap in another off-the-shelf model (which would have the same distribution mismatch problem), I decided to build a Shona-calibrated gender classifier from scratch using the ECAPA embeddings I was already computing for clustering. The key insight is that ECAPA-TDNN embeddings strongly encode gender as a dominant axis of variation — male and female voices form linearly separable regions in the 192-dimensional embedding space. A logistic regression on those embeddings, trained on verified Shona clips, would be both lightweight and calibrated to the actual corpus.

I bootstrapped the training data from the cleanest clusters out of the v2 run — clusters with a single source speaker and unanimous gender votes. That gave me 307 clips across 8 speakers with high-confidence labels. 5-fold cross-validation accuracy was 100%. Then I ran a probe script over all 777 speaker sample clips to validate at scale. The Unknown rate (clips below 65% confidence) was 11.1%. I ear-tested every unknown clip, found most were clearly identifiable, and added ~84 of them to the training data. After retraining, the Unknown rate dropped to 9.4% and the key previously-mislabelled speakers now have zero wrong-gender confident predictions.

This process — extract from clean clusters, train, probe, ear-test unknowns, add to training, retrain — is a lightweight active learning loop that produced a Shona-calibrated gender classifier with no external labelled data and no additional model beyond ECAPA. The two genuinely ambiguous voices (even by ear) were left as Unknown, which is the correct outcome.

The audit pipeline now auto-loads this classifier (`gender_classifier_ecapa.pkl`) and skips the Wav2Vec2 model entirely when it's present. The next step is the v3 audit run, which will use this classifier and — ideally — gender-separated clustering: running HDBSCAN independently within each gender partition so that no cluster can span genders by construction.
